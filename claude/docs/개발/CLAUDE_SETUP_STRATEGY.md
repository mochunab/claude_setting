# Claude Code 범용 세팅 전략서

> `.claude/` 폴더 구조 설계, 레이어별 역할, 프로젝트 문서 작성법
> HARNESS_ENGINEERING.md가 "왜 환경을 통제하는가"라면, 이 문서는 "구체적으로 뭘 어떻게 배치하는가"
> 최종 업데이트: 2026-04-03

---

## 1. .claude/ 폴더 레이어 구조

```
~/.claude/                          ← 글로벌 (모든 프로젝트 공통)
├── CLAUDE.md                       ← 공통 원칙 (언어, 커밋, 작업 방식)
├── rules/                          ← 범용 코딩 표준 (자동 로드)
│   ├── performance.md
│   └── security.md
├── agents/                         ← 범용 서브에이전트
├── skills/                         ← 범용 전문 지식 (트리거 기반 자동 로드)
├── knowledge/                      ← 도메인 지식 라이브러리
│   ├── 개발/
│   ├── 마케팅/
│   └── 비즈니스/
├── commands/                       ← 슬래시 커맨드 (/thread, /property)
├── memory/                         ← 세션 간 기억 (자동 관리)
├── settings.json                   ← 플러그인 + 훅 + 권한
└── settings.local.json             ← 개인 퍼미션 (gitignore)

project/.claude/                    ← 프로젝트별 (오버라이드/확장)
├── CLAUDE.md                       ← 프로젝트 규칙 + 문서 라우팅
├── settings.json                   ← 프로젝트 전용 훅/권한
└── commands/                       ← 프로젝트 전용 워크플로우
```

### 글로벌 vs 프로젝트별 판단 기준

| 질문 | 글로벌 | 프로젝트별 |
|------|--------|-----------|
| 모든 프로젝트에 적용? | O | X |
| 기술 스택에 종속? | X | O |
| 도메인 지식? | 범용(SEO, 보안) | 특화(스코어링 공식) |
| 오버라이드 가능성? | 있음 | 최종 적용 |

## 2. 각 레이어의 역할과 작성 원칙

### CLAUDE.md — 진입점 + 라우팅 테이블

**역할**: 매 세션 자동 로드. 프로젝트의 핵심 원칙 + "상세는 어디서 보라"는 안내.

**작성 원칙**:
- 200줄 이내 (초과 시 AI가 규칙 누락)
- 코드/git에서 유추 가능한 것은 적지 않음
- 하위 문서에 있는 상세를 반복하지 않음 — 링크로 대체
- "주의사항"은 CLAUDE.md에 남김 (gotcha는 매 세션 즉시 필요)

**구조 템플릿**:
```markdown
# 프로젝트명

> 한 줄 설명 + 프로덕션 URL + GitHub

## 프로젝트 개요       ← 3~5줄 핵심만
## 기술 스택           ← 불릿 리스트
## 인프라 요약         ← 2~3줄 + 상세 문서 링크
## 개발 규칙           ← 빌드/배포 커맨드 + 코딩 컨벤션
## 주의사항            ← gotcha (이것만은 매번 기억해야 하는 것)
## 상세 문서           ← 라우팅 테이블 (링크 목록)
## 문서 업데이트 규칙   ← 언제 어떤 문서를 갱신하는지
```

### rules/ — 코딩 표준 자동 적용

**역할**: 코딩 관련 대화에서 자동 로드. CLAUDE.md와 달리 도메인별 분리.

**작성 원칙**:
- 파일당 하나의 관심사 (performance.md, security.md)
- "~하지 마" + "대신 ~해" 쌍으로 작성 (금지만 있으면 AI가 대안을 모름)
- 코드 예시 최소한으로 (규칙 자체가 명확하면 예시 불필요)

### agents/ — 서브에이전트

**역할**: 전문 작업을 메인 컨텍스트에서 분리. 컨텍스트 오염 방지.

**설계 기준**:
- 반복적으로 위임하는 작업 패턴이 보이면 에이전트화
- 에이전트 하나당 하나의 명확한 책임
- 프롬프트에 "출력 형식"을 명시 (에이전트 결과를 메인에서 파싱해야 하므로)

### skills/ — 트리거 기반 전문 지식

**역할**: 특정 키워드/패턴 감지 시 자동 로드. agents와 달리 별도 프로세스가 아니라 메인 컨텍스트에 지식 주입.

**agents vs skills**:
| | agents | skills |
|--|--------|--------|
| 실행 방식 | 별도 프로세스 (컨텍스트 격리) | 메인 컨텍스트에 주입 |
| 용도 | 작업 위임 (코드 리뷰, 배포 점검) | 지식 참조 (DB 패턴, API 규칙) |
| 트리거 | 명시적 호출 또는 CLAUDE.md 워크플로우 | 키워드/import 패턴 자동 감지 |

### knowledge/ — 도메인 지식 라이브러리

**역할**: AI가 "일반 지식"이 아닌 "우리 맥락"으로 답변하게 만드는 레이어.

**작성 원칙**:
- 마크다운 헤딩 + 불릿으로 구조화 (날것의 메모 금지 — 토큰 낭비)
- 하나의 문서 = 하나의 도메인 (SEO, 보안, 성장 전략 등)
- "왜"를 포함 (규칙만 나열하면 AI가 맥락 없이 기계적으로 적용)
- 정기적으로 리뷰 — 오래된 지식은 오히려 해로움

### commands/ — 슬래시 커맨드

**역할**: 반복 프롬프트를 `/커맨드`로 단축 실행.

**만들어야 할 때**: 같은 프롬프트를 3번 이상 복붙하고 있을 때.

### memory/ — 세션 간 기억

**역할**: Claude가 자동 관리. 프로젝트 현황, 피드백, 결정을 세션 간 유지.

**MEMORY.md 관리**:
- 200줄 이내 (초과 시 truncate)
- 한 줄에 하나의 포인터 (`- [제목](파일.md) — 한줄 설명`)
- 오래된/무효한 메모리는 정리

## 3. 프로젝트 문서 작성법

### 문서 분리 기준

하나의 CLAUDE.md에 모든 것을 넣지 않고, 관심사별로 분리.

| 문서 유형 | 역할 | 예시 |
|-----------|------|------|
| OVERVIEW | 현황 요약 + 네비게이션 | Phase 상태 테이블, 핵심 기능 목록 |
| DECISIONS | "왜 이렇게 했는가" 기록 | 설계 배경, 트레이드오프, 근거 |
| FILES | 파일 구조 + 각 파일 역할 | 디렉토리 트리 + 한줄 설명 |
| SCORING/PIPELINE 등 | 도메인 로직 상세 | 공식, 가중치, 데이터 플로우 |
| INFRA | 인프라 구성 상세 | 서버, DB, 크론, 배포 |
| CHECKLIST | 검증 항목 | Phase별 완료/미완료 체크 |
| DESIGN_SYSTEM | UI/UX 토큰 | 컬러, 타이포, 컴포넌트 패턴 |

### DECISIONS.md — 가장 중요한 문서

코드와 git log에서 "무엇을 했는가"는 알 수 있지만, "왜 이렇게 했는가"는 알 수 없음.
DECISIONS가 없으면 다음 세션에서 같은 삽질을 반복하거나 이전 결정을 뒤집음.

**작성 포맷**:
```markdown
## Phase XX — 제목 (날짜)

**배경**: 어떤 상황에서 이 문제를 발견했는가
**문제**: 구체적으로 무엇이 잘못되었는가 (수치 포함)

### 주요 결정
1. **결정 사항**: 무엇을 어떻게 바꿨는가
   - 근거: 왜 이 방식을 선택했는가
   - 기각한 대안: 다른 방법은 왜 안 됐는가 (선택적)

### 영향
- 변경 전 → 변경 후 (수치)
- 영향받는 파일/모듈

### 한계/주의 (선택적)
- 이 결정의 알려진 제약
```

**핵심 규칙**:
- 최신 순 정렬 (위에서부터 읽으면 최근 맥락부터 파악)
- 수치를 넣어라 ("개선됨" 대신 "2,854건 → 108건")
- 기각한 대안도 적어라 (다음 세션이 같은 대안을 다시 시도하는 것을 방지)

### OVERVIEW.md — 세션 시작의 진입점

**작성 원칙**:
- 세션 시작 시 이것만 읽으면 프로젝트 현황 파악 가능
- Phase 상태 테이블 (한 줄 = 한 Phase)
- 상세는 다른 문서로 라우팅 (OVERVIEW 자체는 가볍게 유지)
- 500줄 초과 금지

### 문서 업데이트 트리거

문서를 만들어놓고 갱신하지 않으면 stale 데이터가 되어 오히려 해로움.
CLAUDE.md에 업데이트 규칙을 명시:

```markdown
## 문서 업데이트 규칙

Phase 완료 시 반드시:
1. DECISIONS.md — 설계 결정
2. OVERVIEW.md — 상태 테이블 1줄
3. FILES.md — 파일 목록
4. CHECKLIST.md — 검증 항목

변경 영역별:
- 파이프라인 변경 → PIPELINE.md
- 인프라 변경 → INFRA.md
```

### 단일 문서 크기 기준

- **CLAUDE.md**: 200줄 이내 (매 세션 로드)
- **일반 문서**: 500줄 이내 (초과 시 분리 검토)
- **DECISIONS**: 예외 (히스토리 누적이므로 길어져도 OK, 대신 최신 순)
- **MEMORY.md**: 200줄 이내 (시스템 truncate)

## 4. 점진적 도입 로드맵

한 번에 다 세팅하지 않음. 불편함이 생길 때마다 하나씩 추가.

| 시기 | 세팅 | 해결하는 문제 |
|------|------|-------------|
| Day 1 | CLAUDE.md + rules/ 2개 | 반복 지시 80% 감소 |
| Week 1 | 프로젝트 문서 분리 (OVERVIEW, DECISIONS, FILES) | 세션 간 맥락 단절 |
| Week 1 | commands/ 첫 커맨드 | 반복 프롬프트 자동화 |
| Week 2 | agents/ 커스텀 | 컨텍스트 오염 방지 |
| Week 2 | knowledge/ 도메인 지식 | AI 답변 품질 향상 |
| Week 3 | hooks + settings 최적화 | 안전장치 자동화 |
| 지속 | memory/ 자동 축적 + 정리 | 장기 맥락 유지 |

### 추가 시점 판단법

- "이 설명 또 하네" → knowledge/ 추가
- "이 프롬프트 또 복붙하네" → commands/ 추가
- "이 작업 메인에서 하면 맥락이 날아가네" → agents/ 추가
- "이 규칙 AI가 자꾸 어기네" → rules/ 추가 (또는 린터로 강제 승격)

## 5. 흔한 실수

| 실수 | 왜 문제인가 | 해결 |
|------|-----------|------|
| CLAUDE.md에 모든 걸 넣음 | 200줄 초과 → AI가 규칙 누락 | rules/, knowledge/로 분리 |
| 문서를 만들고 갱신 안 함 | stale 데이터 → 잘못된 결정 유도 | 업데이트 트리거를 CLAUDE.md에 명시 |
| knowledge/에 날것의 메모 | 토큰 낭비 + 노이즈 | 구조화된 마크다운만 허용 |
| DECISIONS에 "무엇"만 기록 | "왜"가 없으면 다음 세션이 뒤집음 | 배경 + 근거 + 기각 대안 필수 |
| 하위 문서 내용을 CLAUDE.md에 반복 | 이중 관리 → 불일치 발생 | 링크로 대체, 요약 2~3줄만 |
| 퍼미션이 200개+ 쌓임 | 관리 불가 | 와일드카드 패턴 활용 (`Bash(npm *)`) |
| 에이전트를 너무 많이 만듦 | 관리 오버헤드 | 반복 패턴이 3회+ 확인된 것만 에이전트화 |

## 6. 문서 품질 체크리스트

문서 작성/수정 후 자가 점검:

- [ ] CLAUDE.md가 200줄 이내인가
- [ ] 하위 문서에 이미 있는 내용을 CLAUDE.md에서 반복하고 있지 않은가
- [ ] DECISIONS에 "왜"가 포함되어 있는가 (배경 + 근거)
- [ ] 수치가 포함되어 있는가 ("개선됨" 대신 구체적 숫자)
- [ ] 500줄 초과 문서가 없는가 (DECISIONS 제외)
- [ ] 문서 업데이트 트리거가 CLAUDE.md에 명시되어 있는가
- [ ] 3개월 이상 갱신 안 된 문서가 없는가
