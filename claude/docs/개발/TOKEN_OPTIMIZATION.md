> 세팅 구조(`CLAUDE_SETUP_STRATEGY.md`)와 하네스 설계(`HARNESS_ENGINEERING.md`)를 전제로 한다.
> 최종 업데이트: 2026-04-03

## 1. 세션 관리 — 가장 기본이자 가장 효과 큰 것

### 핵심 명령어

| 명령어 | 용도 | 언제 쓰는가 |
|--------|------|-------------|
| `/clear` | 대화 히스토리 전체 삭제 | 작업 주제가 완전히 바뀔 때 |
| `/compact` | 기존 대화를 요약 압축 | 같은 주제를 이어가되 컨텍스트가 비대해졌을 때 |

### 세션 분리 원칙

- 하나의 세션 = 하나의 작업 단위 (기능 개발, 버그 수정, 리뷰 등)
- 작업 단위가 바뀌면 `/clear` 후 새로 시작
- 오래된 맥락을 끌고 가면 입력 토큰이 매 턴마다 누적 증가

### resume 시 주의

`claude --resume`으로 세션을 이어가면 이전 대화 전체가 컨텍스트에 올라옴. 대화가 길었던 세션일수록 입력 토큰이 크게 늘어남. 이어갈 맥락이 적다면 새 세션 + 문서 참조가 더 효율적.

## 2. .claudeignore — 불필요한 파일 읽기 차단

### 위치

프로젝트 루트에 `.claudeignore` 파일 생성.

### 권장 설정

```
# 의존성 (대용량)
node_modules/
package-lock.json

# 빌드 결과물
build/
dist/
.next/
.vercel/

# 환경 변수 (보안 + 불필요)
.env
.env.local
.env.production

# 대용량 데이터
*.csv
*.sql
*.db
data/

# 로그
*.log
logs/
```

### 왜 필수인가

Claude Code가 코드 탐색 시 이 파일들을 읽으면 수천~수만 토큰이 소모됨.
`node_modules/`만 차단해도 효과가 체감됨.

## 3. 문서 구조 최적화 — 토큰 낭비 없는 지식 설계

> 상세 구조는 `CLAUDE_SETUP_STRATEGY.md` 참조

### 토큰 관점의 핵심 원칙

| 원칙 | 이유 |
|------|------|
| CLAUDE.md 200줄 이내 | 매 세션 자동 로드 — 길수록 매 턴 토큰 소모 |
| 하위 문서에 있는 내용 반복 금지 | 이중 로드 = 이중 과금 |
| docs/ 파일은 구조화된 마크다운만 | 날것의 메모는 토큰 대비 정보 밀도가 낮음 |
| 일반 문서 500줄 이내 | 초과 시 분리 — 필요한 부분만 로드되게 |

### CLAUDE.md가 길어지면 생기는 일

200줄 초과 시 AI가 규칙을 누락하기 시작함.
400줄 이상이면 매 턴마다 수천 토큰이 "규칙 읽기"에만 소모됨.

**로드 메커니즘 주의**: rules/는 CLAUDE.md와 동일하게 매 세션 항상 로드됨. docs/는 working directory에 등록되어 있으면 탐색 대상이 됨. "관련 대화에서만 자동 로드"되는 것은 skills/ (트리거 기반). 따라서 CLAUDE.md를 짧게 유지하는 것이 핵심이고, rules/로 분리해도 토큰 절약 효과는 없음 — 분리의 목적은 관리 편의성.

## 4. 서브에이전트 활용 — 컨텍스트 격리

### 원리

메인 대화에서 대량의 코드를 읽으면 그 내용이 컨텍스트에 남아 이후 모든 턴의 입력 토큰을 늘림.
서브에이전트는 **별도 컨텍스트**에서 실행되므로 메인 대화가 오염되지 않음.

### 적합한 작업

| 작업 | 에이전트 활용 | 이유 |
|------|-------------|------|
| 코드 리뷰 | O | 대량 diff를 메인에 올리지 않아도 됨 |
| 파일 탐색/검색 | O | 여러 파일을 뒤져야 할 때 |
| 배포 점검 | O | 체크리스트 기반 반복 작업 |
| 단일 파일 수정 | X | 메인에서 바로 하는 게 빠름 |

### 비용 주의

에이전트도 별도 토큰을 소모함. "에이전트를 쓰면 무조건 절약"이 아니라, **메인 컨텍스트 보호가 목적**일 때 효과적.

## 5. Serena 플러그인 — 심볼 단위 코드 탐색

### 원리

파일 전체를 읽는 대신 함수/클래스 단위로 필요한 부분만 조회.
500줄 파일에서 30줄 함수만 읽으면 토큰 소모가 크게 줄어듦.

### 워크플로우

```
1. get_symbols_overview → 파일 구조 파악
2. find_symbol → 특정 함수/클래스 위치
3. find_referencing_symbols → 영향도 분석
4. replace_symbol_body → 코드 수정
```

### 한계

- 인덱싱 안 된 파일은 결국 직접 읽어야 함
- 설정 파일(.yml, .json)은 심볼이 없어서 직접 접근
- MCP 호출 자체도 토큰을 소모
- 파일 줄 수가 적으면(200줄 이하) 직접 읽는 게 나음 — 체감 효과는 파일 크기에 비례

## 6. 모델 선택 — 작업별 비용 차이

| 작업 유형 | 권장 모델 | 이유 |
|-----------|----------|------|
| 간단한 수정, 리팩토링, 포매팅 | Sonnet | 저렴, 충분한 성능 |
| 아키텍처 설계, 복잡한 디버깅 | Opus | 정확도 차이가 비용을 상쇄 |
| 코드 리뷰, 탐색 | Haiku | 단순 반복에 적합. 에이전트에서 쓰려면 `model: haiku` 명시 필요 |

### 비용 절약 습관

- 하나의 세션에서 관련 작업을 연속 처리 (세션을 쪼개면 매번 시스템 프롬프트 + CLAUDE.md 재로드)
- 세션 분리는 "주제가 바뀔 때"이지 "비용 절약"이 아님
- 파일/폴더 범위를 명확히 지정 ("src/utils.ts의 fetchData 함수" vs "이 프로젝트의 API 호출")
- 모호한 질문은 AI가 넓게 탐색함 → 구체적으로 물어볼수록 토큰 절약

## 7. 프롬프트 습관 — 작지만 누적되는 차이

### 하지 말 것

| 습관 | 왜 낭비인가 |
|------|-----------|
| "이 프로젝트 전체 구조 설명해줘" | 모든 파일을 읽으려 함 |
| "모든 파일에서 XXX 찾아줘" | 전체 탐색 → 대량 토큰 |
| 같은 질문을 다른 말로 반복 | 컨텍스트에 중복 누적 |
| 긴 코드를 대화에 직접 붙여넣기 | 이후 모든 턴에서 입력 토큰 증가 |

### 해야 할 것

| 습관 | 효과 |
|------|------|
| 파일 경로 + 함수명 지정 | 정확한 위치만 읽음 |
| "~만 보여줘" 범위 한정 | 불필요한 출력 방지 |
| 에이전트에 탐색 위임 | 메인 컨텍스트 보호 |
| 작업 완료 후 `/compact` | 다음 턴 입력 토큰 절감 |
